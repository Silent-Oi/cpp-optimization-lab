# C++ Optimization Lab

这是一个以 **C++ 性能优化、数值计算和物理问题** 为主线的渐进式练习仓库。
中期目标实现稀疏矩阵，最终目标实现光线追踪。

## 总项目目标

本项目是一个以物理与数值计算问题为载体、以 C++ 性能优化和计算机底层理解为核心的长期学习型项目。项目不会把重点放在单纯实现某个算法或堆叠功能上，而是通过一系列计算过程清晰、性能压力明显、结果可以验证的物理与数学项目，逐步建立从高级程序逻辑到底层硬件执行之间的完整认识。

整个项目从大规模、相互独立的数值计算问题开始，例如批量阻尼振子模拟，随后逐步进入网格计算、偏微分方程、随机模拟、稀疏线性代数、空间数据结构和光线追踪等更复杂的问题。每个子项目既是一个相对完整的物理或数值计算程序，也是一个可控的性能实验平台。通过这些项目，需要逐步理解 C++ 程序中的对象、引用、指针、生命周期、连续内存、数据布局、动态内存分配、缓存局部性、分支预测、编译器优化、自动向量化、多线程并行和任务调度等机制，并能够观察这些机制如何真实地影响程序性能。

项目的核心原则是：先得到简单、清晰、可信的正确实现，再使用性能分析工具找到实际瓶颈，提出明确的优化假设，一次只改变少量因素，重新验证数值结果并测量性能变化。所有优化都必须建立在正确性和可重复性之上，不能仅凭经验猜测某种写法更快，也不能以牺牲数值可靠性、代码可理解性或结果一致性为代价换取表面上的运行速度。

每个子项目都应尽量包含参考实现、优化实现、正确性测试、固定规模的性能测试和实验结论。项目需要记录输入规模、构建模式、硬件环境、运行时间、吞吐量、误差、加速比以及关键实现差异，并能够解释性能变化产生的原因。Git 用于保存开发与优化过程，README 和实验报告用于保存可以长期复现的结论，而不是只保留最终最快的一版代码。

在 C++ 学习方面，本项目的目标不是孤立地学习语言语法，而是在实际数值程序中理解这些语言特性的用途。类和结构体用于组织物理状态和计算模块；引用与指针用于理解数据传递和内存访问；模板用于表达可复用或编译期确定的数值结构；RAII 用于管理资源和对象生命周期；标准容器用于理解容量、分配和连续存储；多线程用于处理可以并行执行的大规模计算任务。所有语言特性的引入都应由项目需求推动，而不是为了使用“高级语法”而使用。

在性能优化方面，本项目需要逐步建立以 CPU 为核心的性能模型，理解程序运行时间不仅由算法复杂度决定，还受到内存访问顺序、缓存命中率、数据结构尺寸、分支行为、浮点运算吞吐量、编译器生成代码、线程同步和内存带宽等因素影响。项目应通过 AoS 与 SoA、连续与非连续存储、按行与按列访问、重复分配与缓冲区复用、单线程与多线程、稠密与稀疏表示、显式矩阵与 matrix-free 方法等对比实验，把这些抽象概念转化为可以测量和解释的结果。

在物理与数学方面，本项目希望将编程作为理解理论的工具。所选问题应尽量具有明确的数学模型、物理意义和验证方式，例如阻尼振子、热传导、波动、粒子输运、Poisson 方程、引力 N 体和光线传播。实现过程中不仅需要关注程序是否运行，还需要理解离散方法、初始条件、边界条件、稳定性、误差、收敛性和计算复杂度。数值结果应尽量与解析解、参考解、守恒量或收敛规律进行比较。

本项目的中期重要目标是完成一个稀疏矩阵与 Poisson 方程求解器。该项目应包括稀疏矩阵的构建和压缩存储、稀疏矩阵向量乘、基本向量运算、共轭梯度法、简单预条件以及二维 Poisson 方程的离散和求解，并比较稠密矩阵、CSR 存储和 matrix-free 方法之间的内存与性能差异。通过这一项目，应建立对稀疏线性代数、迭代求解器、内存访问和科学计算软件结构的基本理解，为后续有限元和高性能求解器方向打下基础。

本项目的最终综合目标是完成一个具备空间加速结构和多线程能力的高性能 CPU 路径追踪器。该项目不仅需要实现光线、相机、几何求交、材质、随机采样和光线多次反弹，还需要实现三角形网格、包围盒、BVH 构建与遍历、图像分块、线程调度和性能分析，并研究几何数据布局、树节点布局、随机数状态、分支行为和并行负载均衡对渲染性能的影响。最终路径追踪器应同时体现数学、物理、算法、数据结构、现代 C++ 和 CPU 优化能力，是整个项目路线的综合成果。

各个子项目不要求一开始就追求高度通用的库设计，也不要求过早加入复杂模板、手写 SIMD、无锁结构、GPU、分布式计算或大型第三方框架。项目优先追求问题边界清晰、实现可理解、结果可验证和性能变化可解释。只有当基础实现、测量方法和瓶颈分析足够可靠后，才进一步尝试自动向量化、手写 SIMD、线程池、更高级的稀疏求解方法或更复杂的渲染算法。

整个项目最终希望形成的能力，不只是“能够写出跑得更快的 C++ 代码”，而是能够面对一个新的计算问题，完成以下过程：建立数学模型，设计数据结构，实现正确版本，构造验证方法，建立性能基准，使用工具定位瓶颈，根据 CPU 和内存执行机制提出优化方案，验证优化效果，并清楚解释速度、内存、精度、复杂度和可维护性之间的权衡。

因此，后续每个子项目的规划都应服务于这个总目标：既完成一个有明确物理或数学意义的计算问题，也引入有限且清晰的新知识，并为后续更复杂的项目提供必要的算法经验、C++ 能力和底层性能理解。项目是否完成，不以代码量或功能数量衡量，而以是否得到正确结果、是否建立可靠测量、是否理解主要性能瓶颈、是否形成可复现结论，以及是否为下一阶段建立了足够基础来衡量。

## 实验原则

进行性能实验时：

- 优先使用 Release 构建。
- 每次只改变一个主要因素。
- 同时检查运行时间和计算结果。
- 至少重复运行数次。
- 记录编译器、参数、规模和硬件环境。
- 不根据一次偶然结果下结论。

实验记录模板见：

- [`docs/learning_log_template.md`](docs/learning_log_template.md)
