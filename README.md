# C++ Optimization Lab

这是一个以物理与数值计算问题为载体、以 C++ 性能优化和计算机底层理解为核心的长期学习仓库。

项目不以代码量、功能数量或“使用了多少高级技术”衡量完成度，而以能否建立可信结果、可靠测量和可解释结论衡量学习成果。

## 长期学习目标

通过一系列计算过程清晰、性能压力明显且结果可以验证的子项目，逐步建立以下能力：

- 把物理或数学问题转化为明确的状态、参数、算法和数据结构。
- 理解离散方法、初始条件、边界条件、稳定性、误差与收敛性。
- 在真实数值程序中理解对象、引用、指针、生命周期、RAII 和标准容器。
- 理解连续内存、数据布局、缓存、分支、向量化、线程调度和内存带宽如何影响 CPU 性能。
- 建立正确性基线、可复现 benchmark，并使用证据判断优化是否有效。
- 解释速度、内存、精度、复杂度、可读性和可维护性之间的权衡。

本项目的中期目标是完成稀疏矩阵与二维 Poisson 方程求解器，掌握 CSR、SpMV、共轭梯度法、简单预条件和 matrix-free 方法。

最终综合目标是完成具备三角形网格、BVH 和多线程能力的高性能 CPU 路径追踪器，把数学、物理、算法、数据结构、现代 C++ 和 CPU 优化能力连接起来。

## 核心学习闭环

每个子项目都应遵循同一条主线：

1. 明确问题边界和数学模型。
2. 实现简单、清晰、可信的正确版本。
3. 建立独立的正确性验证。
4. 建立可复现的性能基准。
5. 使用测量或工具识别实际瓶颈。
6. 提出一个明确的优化假设。
7. 一次只改变一个主要因素，重新验证并测量。
8. 记录结论、限制和可迁移到后续项目的经验。

不得仅凭经验断言某种写法更快，也不得以牺牲数值可靠性或结果一致性换取表面速度。

## 子项目规划契约

后续每个子项目都必须说明：

- **路线定位**：承接了哪些已有能力，将为哪个后续项目提供基础。
- **核心问题**：要完成的物理、数学或算法问题是什么。
- **知识预算**：本项目重点引入哪些有限的新知识，不把所有相关主题一次塞入。
- **范围边界**：核心范围、条件触发的可选实验，以及明确不做的内容。
- **正确性证据**：解析解、参考解、守恒量、收敛规律或交叉实现中的哪一种。
- **性能问题**：准备验证哪个具体性能假设，而不是笼统追求“更快”。
- **里程碑**：每个里程碑的产物、验证证据和需要真正理解的概念。
- **完成标准**：代码、正确性、性能结论和学习总结分别达到什么状态。

规划时遵守以下约束：

- 只详细展开当前里程碑和紧邻的下一里程碑；更远阶段保留粗粒度目标。
- 正确性测试随功能一起建立，不集中拖到项目末尾。
- 在可信 baseline 和瓶颈证据出现前，不进入手动 SIMD、线程池、无锁结构等复杂优化。
- 可选实验必须有明确进入条件；没有证据支持时可以不做。
- 子项目不追求过早通用化。只有至少两个项目需要，或明确属于通用实验基础设施的内容，才进入 `Project00_common`。

新建或重构子项目时使用 [`docs/project_planning_template.md`](docs/project_planning_template.md)。
